Wait... if the unmasking fails, it falls back to the original token as-is!
What if an attacker tries to exploit this fallback?
If they send the exact `signed_cookie_token`, they can bypass masking? Yes, but to do that they must KNOW the `signed_cookie_token`, which defeats BREACH, but BREACH is an attack to steal the token. Allowing the unmasked token defeats BREACH protection if the SPA sends the unmasked token on every request, because then the token is predictable!
Wait, if the SPA reads the cookie (which is unmasked) and sends it in the header, the token in the request header is STATIC for the life of the session!
This completely defeats BREACH protection for the header value! (Though the cookie itself is still static).
But BREACH relies on the token being reflected in the response body. Since the token is only reflected in the body (e.g. in HTML forms) securely via `request.csrf_token`, which is MASKED, the SPA doing `X-CSRF-Token: <static_token>` is perfectly safe, because headers are not vulnerable to BREACH in the same way (unless compressed together, but usually headers and bodies are compressed together? No, HPACK/QPACK is used for HTTP/2/3, and BREACH is specifically against the response body reflecting secrets).
So this is a deliberate and safe feature.
